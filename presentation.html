🐘 למה PostgreSQL לנתונים קבועים?

  • שלמות נתונים (ACID) — זיכוי נקודות ורישום אירוע באותה עסקה; אי-אפשר לזכות בלי תיעוד.
  • אינדקסים בנתיבים החמים — שליפת משתמש או קוד הזמנה ב-O(log n) על כל קריאה.
  • לוג אירועים Append-Only — כל אירוע נרשם פעם אחת ולעולם לא משתנה ⇐ ביקורת מלאה וחישוב מחדש של כל מדד.
💾 דוגמה — נשמר לצמיתות (מקור האמת):
users   → points_balance=350, invite_code="ABC12345"
events  → event_type="attributed", points_delta=+100
users   → last_daily_claim_at=2026-06-12T08:01Z

⚡ למה Redis לנתונים זמניים?

  • תשובה ב-1ms — מטמון יתרה (TTL 15ש') במקום סיבוב מלא ל-DB (~5ms). קריטי באלפי קריאות בשנייה.
  • מונה אטומי — הגבלת קצב נגד הונאה דרך INCR + EXPIRE טבעי ומהיר.
  • עמידות לתקלות — אם Redis נופל, השרת חוזר אוטומטית ל-PostgreSQL בלי לחסום תנועה חיה.
⚡ דוגמה — זמני, נמחק לבד עם TTL:
balance:proj:user_42     → 350    TTL 15s
ratelimit:proj:track:ip → 7      TTL 60s
geoip:88.12.x.91        → "IL"   TTL 24h

🧮 צמיחה ויראלית (SDK)

# users(G) = N₀ × K^G
total(G) = N₀ × (K^(G+1)−1) / (K−1)

→ 28,206 users total

💰 גיוס בתשלום (בסיס)

# constant N₀×K each period
total(G) = N₀ + G × (N₀ × K)
